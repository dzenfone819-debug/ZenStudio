# Visual Style Guide For AI Agents

Этот документ описывает визуальный стиль экрана из текущего проекта так, чтобы другой AI-агент мог сделать любое новое приложение в точно той же эстетике.

## 1. Суть стиля

Это не обычный SaaS UI и не generic dashboard.

Нужная эстетика:
- математический sci-fi интерфейс
- research terminal + control room + cryptographic diagram
- плотный, технический, аналитический экран
- ощущение, что пользователь смотрит не “приложение”, а “операционную карту системы”

Общее настроение:
- темный индиго-космический фон
- теплый янтарный текст
- неоновые акценты по слоям
- тонкие сетки, wireframe-диаграммы, светящиеся ноды
- высокоинформационная плотность
- CRT/scanline/retro-futurism без карикатурности

## 2. Неприкосновенные принципы

Если агент хочет сохранить стиль, он обязан соблюдать это:

1. Основной фон всегда темный, глубокий, многослойный, не плоский.
2. Интерфейс всегда строится как “экран внутри экрана”: большая внешняя рамка со свечением и внутренние аналитические панели.
3. Текст и подписи должны выглядеть инженерно, а не маркетингово.
4. В композиции почти всегда должна быть одна крупная визуальная система слева или в центре:
   диаграмма, карта, сеть, граф, тор, радар, осциллограмма, схема потоков.
5. Справа или снизу должны быть плотные информационные карточки-слои.
6. Акцентные цвета распределяются по смысловым слоям, а не случайно.
7. Скругления крупные, но не “soft startup”; панели должны ощущаться как приборные модули.
8. Не использовать стандартный белый фон, обычные hero-блоки, типичные карточки SaaS, материал-подобные тени, фиолетовый градиент по умолчанию.

## 3. Цветовая система

Использовать эти цвета как первичные токены.

### База

```css
:root {
  --bg: #161338;
  --bg-deep: #0d0a24;
  --panel: rgba(26, 21, 64, 0.88);
  --panel-strong: rgba(38, 31, 86, 0.94);
  --line: rgba(147, 173, 255, 0.22);
  --line-strong: rgba(149, 179, 255, 0.34);
  --text: #f6e7b8;
  --text-dim: #b7b0d8;
  --gold: #ffe08a;
  --orange: #ffa45c;
  --red: #ff678c;
  --cyan: #73f7ff;
  --green: #66f0a4;
  --violet: #d189ff;
  --blue: #5ca2ff;
}
```

### Правила использования цветов

- `--text` и `--gold`: основная типографика, заголовки, лейблы, формулы.
- `--text-dim`: вторичные описания, технические примечания, поясняющие строки.
- `--red`: compute / urgent / power / first layer.
- `--orange`: intelligence / economic transform / pricing.
- `--green`: protocol / verified flow / connectivity.
- `--cyan`: value node / active signal / technical highlights.
- `--violet`: applications / abstract systems / outer layer.
- `--blue`: связующие интерфейсные линии и вспомогательные свечения.

### Цветовые запреты

- Не заменять основной фон на чистый черный.
- Не использовать чисто белые карточки.
- Не уводить палитру в кислотную synthwave-неонщину.
- Не делать акценты слишком равномерными: должен быть один теплый золотой текстовый слой и несколько локальных неоновых слоев.

## 4. Типографика

### Шрифтовая пара

- Заголовки и технические ярлыки: `Orbitron`
- Основной интерфейсный текст, подписи, числа, легенды: `Azeret Mono`

Fallback:
- `"Orbitron", "Azeret Mono", monospace`
- `"Azeret Mono", "IBM Plex Mono", "SFMono-Regular", monospace`

### Типографические правила

- Заголовки почти всегда uppercase.
- Межбуквенный интервал заметный: `0.08em` - `0.18em`.
- Подписи короткие, сухие, технические.
- Текст не должен звучать как реклама.
- Хорошо работают формулы, индексы, символы, сокращения, маркировка слоев `L1`, `L2`, `T²`, `π₁`, `MCP`.

## 5. Фон и атмосфера

Фон должен быть многослойным:
- глубокий индиго-градиент
- локальные цветовые облака через `radial-gradient`
- едва заметная инженерная сетка
- поверх всего легкий CRT/scanline-слой

Обязательные атмосферные элементы:
- мягкое свечение у краев
- почти незаметный шум или scanline-слой
- сетка крупного шага, примерно `72px - 80px`
- панель-оболочка с большой рамкой и тенью

Нельзя:
- оставлять ровный однотонный фон
- делать “чистый corporate minimal”
- использовать glassmorphism как главный прием

## 6. Контейнер и раскладка

### Внешний shell

Характеристики:
- крупная внешняя рамка
- радиус `24px - 30px`
- тонкий полупрозрачный бордер
- массивная тень
- внутренний padding `16px - 20px`
- ощущение “панели прибора”

### Предпочтительная композиция

Базовая сетка:
- 3 колонки
- левая колонка: главный визуальный объект
- центральная колонка: связки, потоки, мини-диаграммы
- правая колонка: стек аналитических карточек и метрик

Если экран другой по смыслу, композицию можно менять, но надо сохранить иерархию:
- один крупный system-visual
- несколько плотных data-panels
- горизонтальные/вертикальные сигнальные связи между ними

## 7. Панели и карточки

Каждая панель должна:
- иметь темный полупрозрачный фон
- тонкий светящийся бордер
- легкий внутренний highlight
- аккуратное скругление `16px - 18px`
- ощущение физического модуля

Не делайте:
- обычные flat cards
- толстые белые бордеры
- большие пустые области внутри карточек

Панели лучше наполнять:
- короткими техническими заголовками
- двухстрочными описаниями
- числами, символами, микро-легендами
- локальными индикаторами или мини-визуализациями

## 8. Графический язык

Это ключевая часть стиля.

Используйте:
- wireframe-формы
- диаграммы потоков
- светящиеся точки-ноды
- тонкие горизонтальные сигнальные линии
- мини-графики внутри карточек
- математические обозначения
- легенды с цветными маркерами

Особенно уместны:
- торы, кольца, орбиты, графы
- сетки и траектории
- схемы слоев
- прогресс-бары инженерного типа
- “операционная карта” вместо иллюстрации

Избегайте:
- стоковых иконок
- скругленных blob-форм
- баннерных иллюстраций
- generic charts из BI-панелей

## 9. Компоненты, которые хорошо переносят стиль

Этот стиль хорошо ложится на любые приложения, если адаптировать их в такие компоненты:

- layer stack
- system map
- signal field
- benchmark bars
- technical legend
- theorem/equation block
- control tabs
- node timeline
- matrix table
- protocol cards

Даже если это CRM, email-клиент или editor, переводите сущности в этот язык:
- проекты -> layers
- задачи -> signals
- пользователи -> nodes
- статусы -> illuminated markers
- workflow -> cascade

## 10. Motion

Анимация должна быть живой, но очень сдержанной.

Разрешено:
- медленный pulse у нод
- слабый flicker у сигналов и индикаторов
- микро-drift у траекторий
- едва заметные переходы прозрачности и свечения

Запрещено:
- прыгающие карточки
- heavy parallax
- пружинные UI-анимации как в мобильных лендингах
- постоянное движение крупных блоков

Motion должен ощущаться как:
- работающий терминал
- активная схема
- прибор, который “жив”, но не отвлекает

## 11. Информационная плотность

Нужен высокий signal-to-noise ratio.

Правила:
- на экране должно быть много информации, но она должна быть четко структурирована
- мало длинных абзацев
- много коротких строк, лейблов, чисел, кодовых обозначений
- каждый блок должен выглядеть содержательно
- пустоты допустимы только вокруг главной диаграммы

## 12. Адаптация под новые приложения

Чтобы сохранить стиль в любой новой задаче:

1. Сначала определить “главную систему” приложения.
2. Превратить ее в крупную диаграмму.
3. Остальные сущности разложить по карточкам слоев.
4. Добавить легенду, сигналы, мини-графики, статусные ноды.
5. Сохранить палитру, шрифты и плотность.

Примеры:
- todo app -> task cascade terminal
- finance app -> clearing house dashboard
- AI chat -> multi-agent signal console
- CMS -> content topology map
- analytics -> protocol intelligence board

## 13. Do / Don't

### Do

- Делать интерфейс как научно-операционный экран.
- Использовать индиго-фон + золото + локальные неоновые акценты.
- Сохранять крупную диаграмму и плотные data-panels.
- Писать короткие инженерные заголовки.
- Добавлять технические микро-детали.
- Использовать SVG, CSS grid, layered gradients, subtle glow.

### Don't

- Не делать обычный landing page.
- Не использовать типичный SaaS UI-kit.
- Не скатываться в cyberpunk arcade.
- Не перегружать яркими цветами весь экран сразу.
- Не делать большие белые участки.
- Не заменять диаграмму на random hero-illustration.

## 14. Минимальный набор токенов для нового проекта

Если агент делает новый экран с нуля, у него должны быть как минимум:

- темный layered background
- outer shell container
- panel component
- heading style
- mono body text
- gold labels
- 5 accent colors for semantic layers
- scanline overlay
- grid overlay
- one major SVG/system diagram
- one stack of technical cards
- one legend
- one status/benchmark block

## 15. Базовый промпт для другого AI-агента

Можно использовать как стартовую инструкцию:

> Create the interface in the exact same visual language as the reference ZenStudio screen: dense retro-futurist analytical dashboard, deep indigo layered background, amber technical typography, monospaced systems labels, Orbitron headers, glowing semantic layer cards, thin grid overlays, subtle CRT scanlines, one large system diagram as the main visual anchor, and supporting technical panels with equations, legends, signal lines, node markers, and benchmark bars. Avoid generic SaaS layouts, white surfaces, soft startup aesthetics, and cartoon cyberpunk. The UI must feel like a mathematical control console, not a marketing page.

## 16. Правило точности

Если новый экран должен быть “в том же стиле”, агент обязан в первую очередь повторить:
- палитру
- шрифтовую пару
- тип рамок и карточек
- плотность информации
- графический язык диаграмм и сигналов
- атмосферный фон

Если эти 6 пунктов не совпадают, значит стиль не сохранен.

## 17. Практическая реализация

Для web-реализации предпочтительны:
- HTML + CSS Grid/Flex
- SVG для главных диаграмм
- CSS gradients для фона
- CSS pseudo-elements для glow/grid/scanlines
- минимальная анимация через CSS keyframes

Не стоит в первую очередь опираться на:
- готовые dashboard templates
- chart libraries с дефолтной темой
- стандартные component kits без кастомизации

## 18. Краткая проверка результата

Перед сдачей агент должен проверить:

1. Экран сразу читается как один и тот же визуальный мир?
2. Есть ли большой system-visual, а не просто набор карточек?
3. Палитра осталась индиго + золото + неоновые слои?
4. Есть ли инженерная плотность и техническая типографика?
5. Нет ли ощущения generic SaaS, landing page или fintech template?

Если хотя бы на один вопрос ответ “нет”, стиль нужно дорабатывать.
